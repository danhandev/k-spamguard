# 경쟁 포지셔닝: K-SpamGuard 차별화 전략
**버전 0.1 · 2026-05-05**

---

## 1. 경쟁 구도 요약

| 구분 | Vybbi류 (자동화 우선) | **K-SpamGuard (신뢰 우선)** |
|---|---|---|
| 핵심 약속 | "연결하면 알아서 다 지워줌" | "왜 지우는지 알고, 실수해도 돌릴 수 있음" |
| 탐지 방식 | 블랙박스 (이유 비노출) | 정규화 텍스트 + 매칭 룰 + 스코어 공개 |
| 기본 액션 | 즉시 삭제 | 숨김(hide) 우선, 삭제는 명시적 선택 |
| 오탐 대응 | 없거나 불명확 | 1클릭 복구 (숨김 해제), UNCERTAIN 범위 검수 큐 |
| 과금 단위 | 삭제 건수 기반 | 월정액 처리 한도 (예측 가능) |
| 감사 로그 | 없음 | 불변 Audit Log (모든 액션 기록) |
| 타겟 | 빠른 처리 원하는 대형 계정 | 브랜드 신뢰도가 중요한 중소형 계정 |

## 2. 고객 세그먼트 차별화

### Vybbi류가 잘 맞는 고객
- 팔로워 100만+, 댓글 수천 개/일
- 오탐 1~2건쯤 감수하고 속도가 중요한 계정
- 스팸 삭제를 "인프라 비용"으로 취급하는 대형 MCN

### K-SpamGuard가 잡아야 할 고객
- **팔로워 1만~50만, 댓글 수백 개/일** — 자동 삭제 실수가 브랜드 이미지에 직접 영향
- **쇼핑몰/뷰티 브랜드** — 고객 댓글을 잘못 지우면 CS 사고
- **크리에이터 개인** — 팬 댓글을 스팸으로 오판할까봐 자동화를 두려워하는 사람
- **감사/증거 기반 운영이 필요한 사업자** — "왜 이 댓글을 지웠는지" 설명할 수 있어야 함

> **핵심 인사이트**: Vybbi는 "스팸 처리의 귀찮음"을 판다. K-SpamGuard는 "실수 없는 운영"을 판다. 같은 문제를 사는 고객이 다르다.

---

## 3. 가격 포지션

### Vybbi류의 구조적 문제
- 삭제 건수 기반 과금 → **사용량이 늘수록 비용 불안**
- 스팸이 폭주하는 날(광고 게시일, 바이럴 당일) 요금이 급증
- 사용자가 "오늘 얼마 나올까" 계산하게 만듦

### K-SpamGuard 포지션
```
월정액 처리 한도 방식
→ "이번 달 최대 얼마" 예측 가능
→ 스팸 폭주 날도 추가 요금 없음
→ 한도 초과 시 액션 차단 (요금 추가 청구 X)
```

| 플랜 | 요금 | 포지션 |
|---|---|---|
| Free | 무료 | 진입 장벽 제거. "써보고 결정" |
| Starter (9,900원) | 커피 2잔 값 | 개인 크리에이터의 심리적 저항 최소화 |
| Pro (49,900원) | 쇼핑몰 운영 비용의 극히 일부 | "직원 30분/일 아끼는 것보다 쌈" |

**피해야 할 포지션**: "업계 최저가" — 신뢰도 있는 B2B SaaS는 최저가로 싸우지 않는다.

## 4. 한국어 스팸 탐지 차별화

Vybbi류는 탐지 로직을 공개하지 않는다. 이것이 약점이다.

### K-SpamGuard의 탐지 투명성

**댓글 카드에 표시하는 정보:**
```
원문:    "카·지·노 ㅅㅍㅇ 0nlyfans 클릭→ bit.ly/xxx"
정규화:  "카지노 스팸 onlyfans 클릭 bit.ly/xxx"
스코어:  0.94 (임계값 0.7 초과)
매칭 룰: [도박-키워드: "카지노" +0.4]
          [초성패턴: "ㅅㅍㅇ" +0.3]
          [URL포함: "bit.ly" +0.2]
          [성인키워드: "onlyfans" +0.04]
```

**왜 이게 차별화인가:**
1. 사용자가 왜 스팸으로 잡혔는지 납득 → 자동 모드에 대한 신뢰 형성
2. 오탐 발견 시 "어떤 룰이 문제인지" 바로 파악 → 룰 조정 가능
3. "AI가 알아서" 블랙박스 vs "이 규칙 때문에" 투명한 판단

### 한국어 변형 탐지 기술 (Vybbi가 없는 것)
| 변형 유형 | K-SpamGuard 처리 | 경쟁사 |
|---|---|---|
| 초성 분리 (ㄷㅂ, ㅅㅍㅇ) | 초성 패턴 → 단어 복원 후 매칭 | 대부분 미지원 |
| 특수문자 삽입 (카·지·노) | NFKC 정규화 + 특수문자 제거 | 일부 지원 |
| 숫자 치환 (0→o, 1→l) | 문자 치환 테이블 적용 | 대부분 미지원 |
| 전각/반각 혼용 (Ｃａｓｉｎｏ) | Unicode NFKC 정규화 | 일부 지원 |
| 단축 URL | URL 추출 + 도메인 블랙리스트 | 일부 지원 |

> **포트폴리오 관점**: 이 파이프라인이 존재한다는 것 자체가 "한국어 도메인 이해"를 증명. 단순 키워드 필터를 구현한 것이 아님을 보여준다.

## 5. 안전한 삭제 UX

### Vybbi류의 UX 문제
- "삭제됨" 상태로 끝. 되돌릴 방법 없음.
- 오탐이 발생해도 사용자가 인지할 시점엔 이미 삭제됨.

### K-SpamGuard의 3단계 안전망

```
[단계 1] 탐지 → 검수 큐 (사용자가 보고 결정)
                    ↓
[단계 2] 승인 → 숨김(hide) 먼저
               → 팔로워에게는 안 보이지만 DB엔 존재
                    ↓
[단계 3] 명시적 삭제 선택 시에만 DELETE 호출
               → "정말 삭제하시겠습니까?" 확인 없음
                 (숨김이 이미 안전장치이므로 UX 마찰 최소화)
```

**오탐 발견 시 복구 플로우:**
```
Audit Log → "자동 처리된 댓글" 탭 → 복구 버튼
→ Instagram API hidden=false 호출
→ 댓글 다시 노출
→ Audit Log: restore_action 기록
```

**왜 숨김 우선인가:**
- Instagram의 숨김 API는 되돌릴 수 있음 (`hidden=false`)
- 삭제 API는 복구 불가
- 사용자가 자동 모드를 믿으려면 실수를 돌릴 수 있어야 함

## 6. 금융권 포트폴리오로 보여줄 기술 요소

Vybbi류 서비스는 "동작하는 제품"을 보여준다.
K-SpamGuard는 "신뢰할 수 있는 시스템"을 보여준다.

### 채용 담당자가 보는 포인트

| 기술 요소 | 구현 방식 | 금융권 연결 |
|---|---|---|
| **불변 감사 로그** | audit_logs 테이블 INSERT ONLY, 별도 트랜잭션 | 금융 거래 로그, 규제 대응 (ISMS) |
| **OAuth 토큰 암호화** | AES-256-GCM + per-token IV | 개인정보보호법, 금융보안원 기준 |
| **멱등성 보장** | Redis idempotency key + DB UNIQUE 제약 | 금융 이중 출금 방지 패턴 |
| **Circuit Breaker** | Resilience4j, 슬라이딩 윈도우 | 대외 API 장애 격리, 카드사 API 패턴 |
| **외부 ID 분리** | 내부 Long PK ↔ 외부 UUID 분리 | IDOR 방어, 금융 계정 노출 방지 |
| **Hexagonal Architecture** | Port 인터페이스 → 어댑터 교체 가능 | 유지보수성, 테스트 가능성 |
| **Testcontainers** | 실제 PostgreSQL 대상 통합 테스트 | 운영 DB 환경 재현, 마이그레이션 검증 |

**포트폴리오 서술 방식:**
> "Instagram API의 불안정성을 신뢰할 수 없는 외부 의존성으로 간주하고, Circuit Breaker / Retry / Idempotency 조합으로 99% 이상의 액션 성공률을 달성했습니다. 이 패턴은 카드사 API, 코어뱅킹 연동에서도 동일하게 적용됩니다."

## 7. 출시 순서 전략

### 왜 순서가 중요한가
Vybbi는 "자동 삭제"를 첫날부터 제공한다. 그래서 빠르지만, 초기 오탐으로 신뢰를 잃으면 이탈이 크다.
K-SpamGuard는 신뢰를 먼저 쌓고 자동화를 나중에 제공한다.

```
Phase 1: 검수 큐 + 숨김 (수동 모드만)
  → 사용자가 직접 탐지 품질을 확인
  → "이 서비스가 정확하다"는 신뢰 형성
  → 오탐률을 사용자 데이터로 측정

Phase 2: 자동 모드 (threshold 설정 후 선택적 활성화)
  → "신뢰가 생긴 사람만" 자동화로 업그레이드
  → 유료 플랜 전환 동인

Phase 3: 삭제 (숨김 충분히 검증된 후)
  → 숨김으로 운영하다가 "이제 바로 지워도 되겠다"는 판단이 선 사용자에게 제공
```

**경쟁사와 역방향**: Vybbi는 자동 → 수동 복구. K-SpamGuard는 수동 → 자동 확장.
이 역방향이 "신중한 B2B SaaS"의 포지션을 만든다.

## 8. 피해야 할 기능 및 메시지

### 기능 측면
| 피해야 할 것 | 이유 |
|---|---|
| "연결하면 바로 자동 삭제" | Vybbi와 동일. 오탐 리스크. 신뢰 구축 전 제공 위험 |
| 삭제 건수 기반 과금 | 사용자 불안 조성. 스팸 폭주 날 이탈 유발 |
| "AI 기반 탐지" 마케팅 | 룰 엔진인데 AI로 포장하면 설명 책임 생김. 오탐 시 신뢰 붕괴 |
| 블랙박스 처리 (이유 미제공) | 투명성이 차별화인데 이를 포기하는 것 |
| 복구 없는 즉시 삭제 | 오탐 = 사용자 이탈. 초기에는 특히 위험 |
| 과도한 소셜 분석 기능 | 팔로워 분석, 해시태그 트래킹 → Vybbi와 다른 제품이 됨 |

### 메시지 측면
| 피해야 할 표현 | 대신 사용 |
|---|---|
| "완전 자동화" | "검수 후 자동화, 실수는 복구" |
| "AI가 다 알아서" | "어떤 룰에 걸렸는지 스코어로 확인" |
| "경쟁사보다 싸다" | "예측 가능한 월정액, 폭주 당일도 추가 요금 없음" |
| "스팸 100% 차단" | "한국어 변형 패턴 탐지 Precision 90% 이상" |
